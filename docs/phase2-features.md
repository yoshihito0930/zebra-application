# Phase 2以降の実装予定機能

このドキュメントは、MVP（Phase 1）では実装を見送り、Phase 2以降に実装予定の機能をまとめたものです。

**最終更新日**: 2026-04-13

---

## Phase 1（MVP）で実装した機能

Phase 1では、予約管理システムの**最小限の機能**を実装しました:

### 実装済みエンドポイント（11個）

#### 顧客向け
1. POST /auth/signup - ユーザー登録
2. POST /auth/login - ログイン
3. GET /studios/{id}/calendar - カレンダー表示（予約状況確認）
4. GET /studios/{id}/plans - プラン一覧取得
5. POST /reservations - 予約作成
6. GET /reservations/me - 自分の予約一覧
7. GET /reservations/{id} - 予約詳細
8. PATCH /reservations/{id} - 予約内容編集
9. PATCH /reservations/{id}/cancel - 予約キャンセル

#### 管理者向け
10. GET /reservations - 予約一覧（管理用）
11. PATCH /reservations/{id}/approve - 予約承認

### 実装済みバッチ処理（1個）
- 仮予約期限切れ処理（日次、EventBridge + Lambda）

## Phase 2以降に実装する機能

以下の機能は、MVP検証後に順次実装します。

### 1. 予約関連機能の拡張

#### 1.1 予約拒否機能
**エンドポイント**: PATCH /reservations/{id}/reject

**概要**:
- 管理者が予約リクエストを拒否できる機能
- 拒否理由を入力してステータスを `cancelled` に変更
- 拒否理由は顧客にメール通知される

**リクエスト例**:
```json
{
  "reason": "設備メンテナンスのため"
}
```

**ビジネス要件**:
- pending ステータスの予約のみ拒否可能
- 拒否理由は任意だが、500文字以内
- 拒否時に顧客へメール通知（SES使用）

**参照**: [docs/api-design.md](api-design.md) の「PATCH /reservations/{id}/reject」セクション

---

#### 1.2 仮予約→本予約切り替え機能
**エンドポイント**: PATCH /reservations/{id}/promote

**概要**:
- 顧客が仮予約（tentative）を本予約（regular）に切り替える機能
- 切り替え後、ステータスは `pending` になり、管理者の承認待ちになる

**ビジネス要件**:
- tentative ステータスの予約のみ昇格可能
- 昇格後は `promoted_from: "tentative"` を記録
- 昇格時に時間帯の重複チェックを再実行（他の予約が入っている場合はエラー）

**参照**: [docs/api-design.md](api-design.md) の「PATCH /reservations/{id}/promote」セクション

---

### 2. プラン・オプション管理機能

#### 2.1 プラン管理API
**エンドポイント**:
- POST /plans - プラン作成
- PATCH /plans/{id} - プラン更新

**概要**:
- 管理者がプランを作成・更新できる機能
- 料金変更、プランの有効化/無効化が可能

**MVP期間の代替手段**:
- DynamoDBコンソールから直接プランデータを登録
- または、AWS CLIで `PutItem` を実行

**ビジネス要件**:
- プラン名は1〜100文字
- 料金は0以上の整数
- 税率は0〜1の小数（例: 0.10）
- プランを無効化（`is_active: false`）しても、既存の予約には影響しない（スナップショット保存されているため）

**参照**: [docs/api-design.md](api-design.md) の「POST /plans」「PATCH /plans/{id}」セクション

---

#### 2.2 オプション管理API
**エンドポイント**:
- GET /studios/{id}/options - オプション一覧取得（※実装済み）
- POST /options - オプション作成
- PATCH /options/{id} - オプション更新

**概要**:
- 管理者がオプションを作成・更新できる機能
- 追加料金の設定、オプションの有効化/無効化が可能

**MVP期間の代替手段**:
- プラン管理と同様、DynamoDBコンソールまたはAWS CLIで直接登録

**参照**: [docs/api-design.md](api-design.md) の「POST /options」「PATCH /options/{id}」セクション

### 3. ブロック枠管理機能

**エンドポイント**:
- POST /blocked-slots - ブロック枠作成
- GET /blocked-slots - ブロック枠一覧取得
- DELETE /blocked-slots/{id} - ブロック枠削除

**概要**:
- 管理者が特定の日時を予約不可に設定できる機能
- 定休日、設備メンテナンス、プライベート利用などで使用

**MVP期間の代替手段**:
- DynamoDBコンソールから直接ブロック枠データを登録

**ビジネス要件**:
- 全日ブロック（`is_all_day: true`）または時間指定ブロック
- ブロック枠が設定されている時間帯は、予約作成時にエラーになる
- 理由は必須（1〜200文字）

**参照**: [docs/api-design.md](api-design.md) の「ブロック枠」セクション

---

### 4. 問い合わせ管理機能

**エンドポイント**:
- POST /inquiries - 問い合わせ作成
- GET /inquiries/me - 自分の問い合わせ一覧
- GET /inquiries - 問い合わせ一覧（管理用）
- PATCH /inquiries/{id}/reply - 問い合わせ回答

**概要**:
- 顧客がスタジオに問い合わせを送信できる機能
- 管理者が問い合わせに回答し、顧客にメール通知

**MVP期間の代替手段**:
- 顧客はメール（例: info@studiozebra.com）で問い合わせ
- 管理者はメールクライアントで返信

**ビジネス要件**:
- 問い合わせタイトルは1〜100文字
- 問い合わせ内容は1〜2000文字
- ステータス: open（未回答）、replied（回答済み）、closed（クローズ）

**参照**: [docs/api-design.md](api-design.md) の「問い合わせ」セクション

---

### 5. ユーザー管理機能

#### 5.1 プロフィール管理
**エンドポイント**:
- GET /users/me - 自分のプロフィール取得
- PATCH /users/me - 自分のプロフィール更新

**概要**:
- ユーザーが自分のプロフィール（電話番号、住所など）を更新できる機能

**MVP期間の代替手段**:
- ユーザー情報の変更は管理者に依頼（メールやSlackで連絡）

**ビジネス要件**:
- email、role の変更は不可（セキュリティ上の理由）
- 電話番号、会社名、住所のみ変更可能

**参照**: [docs/api-design.md](api-design.md) の「PATCH /users/me」セクション

---

#### 5.2 ユーザー一覧・詳細取得（管理用）
**エンドポイント**:
- GET /users/{id} - ユーザー詳細取得
- GET /users - ユーザー一覧取得

**概要**:
- 管理者が顧客情報を閲覧できる機能

**MVP期間の代替手段**:
- DynamoDBコンソールから直接usersテーブルを参照

**参照**: [docs/api-design.md](api-design.md) の「ユーザー」セクション

---

#### 5.3 スタッフ管理
**エンドポイント**:
- POST /staff - スタッフユーザー登録
- GET /staff - スタッフ一覧取得

**概要**:
- 管理者がスタッフアカウントを作成できる機能
- スタッフは予約の閲覧のみ可能（編集・承認はできない）

**MVP期間の代替手段**:
- スタッフが必要な場合、管理者アカウントを共有（推奨しないが、MVP期間のみ）

**ビジネス要件**:
- スタッフは `role: "staff"` で作成
- 所属スタジオ（`studio_id`）を紐付ける
- スタッフは予約の閲覧のみ可能（POST/PATCH操作は403エラー）

**参照**: [docs/api-design.md](api-design.md) の「POST /staff」「GET /staff」セクション

### 6. バッチ処理の拡張

Phase 1では「仮予約期限切れ処理」のみ実装しましたが、以下のバッチ処理を追加します。

#### 6.1 仮予約期限通知（日次）
**実行タイミング**: 毎日深夜（例: 午前2時）

**処理内容**:
1. 利用日まで3日（設定可能）の仮予約を取得
2. 顧客にメール通知「仮予約の期限が近づいています。本予約に切り替えるか、キャンセルしてください」
3. notificationsテーブルに通知履歴を記録

**ビジネス要件**:
- 同じ予約に対して1回のみ通知（重複送信防止）
- メールテンプレートはSESで管理

**参照**: [docs/requirements.md](requirements.md) の「バッチ処理」セクション

---

#### 6.2 第2キープ繰り上げ処理（イベント駆動）
**実行タイミング**: 第1候補（本予約/仮予約）がキャンセルされた時

**処理内容**:
1. キャンセルされた予約の日時・スタジオIDを取得
2. 同一時間帯の第2キープ（`status: "waitlisted"`）を検索
3. 第2キープを `tentative`（仮予約）に昇格
4. 顧客にメール通知「第1候補がキャンセルされたため、仮予約に繰り上げされました」

**実装方法**:
- DynamoDB Streamsを使用してキャンセルイベントを検知
- または、予約キャンセルAPI内で同期的に実行

**ビジネス要件**:
- 第2キープが複数ある場合、最初に作成されたものを優先
- 繰り上げ後の有効期限は利用日の7日前まで（通常の仮予約と同じ）

**参照**: [docs/requirements.md](requirements.md) の「第2キープ」セクション

---

#### 6.3 リマインド通知（日次）
**実行タイミング**: 毎日深夜（例: 午前2時）

**処理内容**:
1. 翌日の確定予約（`status: "confirmed"`）を取得
2. 顧客にリマインドメール「明日の予約のリマインドです」
3. 予約詳細（日時、プラン、オプションなど）を含める

**ビジネス要件**:
- メールテンプレートにはスタジオの住所、アクセス情報を含める
- 通知済みフラグをnotificationsテーブルに記録

**参照**: [docs/requirements.md](requirements.md) の「バッチ処理」セクション

---

#### 6.4 予約完了処理（日次）
**実行タイミング**: 毎日深夜（例: 午前3時）

**処理内容**:
1. 利用日が過ぎた予約（`date < 今日`）を取得
2. ステータスを `completed` に更新

**ビジネス要件**:
- `cancelled`、`expired` ステータスの予約は対象外
- 完了処理後、レビュー依頼メールを送信（オプション）

**参照**: [docs/requirements.md](requirements.md) の「バッチ処理」セクション

---

### 7. 認証方式の移行（Cognito統合）

**概要**:
Phase 1ではモック認証を使用していますが、本番環境ではAWS Cognitoに移行します。

**移行タイミング**:
- ユーザー数が50,000人に近づいたとき（Cognito無料枠の上限）
- または、セキュリティ要件が厳しくなったとき（MFA、パスワードポリシー強化など）

**移行手順**:
詳細は [docs/migration-plan.md](migration-plan.md) を参照してください。

## Phase 2以降の実装優先順位

以下の順序で実装することを推奨します:

### フェーズ2A（運用効率化）
1. **プラン・オプション管理API** - DynamoDB直接操作から脱却
2. **ブロック枠管理API** - 定休日やメンテナンス日の設定を容易に

### フェーズ2B（顧客体験向上）
3. **リマインド通知バッチ** - 予約忘れを防ぐ
4. **仮予約期限通知バッチ** - 仮予約の期限切れを防ぐ
5. **仮予約→本予約切り替え機能** - 顧客の利便性向上

### フェーズ2C（高度な予約管理）
6. **第2キープ繰り上げ処理** - 第2キープ機能を完全に活用
7. **予約拒否機能** - 管理者が柔軟に予約を管理

### フェーズ2D（サポート機能）
8. **問い合わせ管理機能** - 顧客とのコミュニケーションを一元管理
9. **ユーザープロフィール管理** - 顧客が自分で情報を更新
10. **スタッフ管理機能** - スタッフアカウントの権限分離

### フェーズ2E（セキュリティ強化）
11. **Cognito統合** - 本番環境での認証セキュリティ強化

## Phase 2実装時の注意点

### 1. データベーススキーマの互換性
Phase 1で作成した予約データは、Phase 2でも引き続き使用します。新機能追加時は以下に注意:
- 既存フィールドの削除は避ける（後方互換性を保つ）
- 新しいフィールドを追加する場合、既存データに対してデフォルト値を設定

### 2. API仕様の後方互換性
既存のエンドポイントのレスポンス形式を変更する場合:
- 新しいフィールドを追加するのはOK
- 既存フィールドの削除や型変更は避ける（フロントエンドが壊れる）
- どうしても変更が必要な場合は、APIバージョニング（例: /v2/reservations）を検討

### 3. バッチ処理の実装順序
バッチ処理は以下の順序で実装することを推奨:
1. リマインド通知（顧客体験向上、実装が簡単）
2. 仮予約期限通知（期限切れ防止、実装が簡単）
3. 第2キープ繰り上げ処理（複雑だが、ビジネス価値が高い）
4. 予約完了処理（優先度低、手動でも対応可能）

### 4. Cognito移行のリスク
モック認証からCognitoへの移行時は、以下のリスクに注意:
- 既存ユーザーのパスワード移行が必要（Cognitoのインポート機能を使用）
- トークン形式が変わるため、フロントエンドの認証ロジックも修正が必要
- 移行期間中は両方の認証方式をサポートする「ハイブリッド認証」を検討
