# デザイン設計

## デザインコンセプト

スタジオゼブラ予約管理アプリケーションは、撮影スタジオという創造的な空間にふさわしい、洗練されたデザインを目指します。

### 基本方針

- **クリーン & ミニマル**: 無駄を削ぎ落とし、本質的な情報と操作に集中できるデザイン（Apple, Adobe系）
- **プロフェッショナル**: クリエイター向けツールとしての信頼感と品質
- **直感的**: 初めてのユーザーでも迷わず使えるUI/UX
- **視覚的明快さ**: カレンダーと予約状態を一目で把握できる視覚設計
- **レスポンシブ**: デスクトップ・タブレット・モバイルすべてに対応

## UI/UX方針

### レスポンシブデザイン

#### ブレイクポイント
```css
/* Mobile First */
--breakpoint-sm: 640px;   /* タブレット縦 */
--breakpoint-md: 768px;   /* タブレット横 */
--breakpoint-lg: 1024px;  /* デスクトップ */
--breakpoint-xl: 1280px;  /* 大画面デスクトップ */
```

#### デバイス別レイアウト
- **モバイル（〜640px）**: シングルカラム、フルスクリーンモーダル
- **タブレット（640px〜1024px）**: 一部2カラム、モーダルは80%幅
- **デスクトップ（1024px〜）**: 2〜3カラム、サイドバー表示、モーダルは固定幅

### アクセシビリティ（WCAG 2.1 AA準拠目標）

1. **カラーコントラスト**: テキストと背景のコントラスト比 4.5:1 以上
2. **フォーカス表示**: キーボード操作時の明確なフォーカスリング
3. **ラベル**: すべてのフォーム要素に適切なラベル
4. **ARIAラベル**: スクリーンリーダー対応
5. **キーボード操作**: すべての機能をキーボードで操作可能
6. **色のみに依存しない**: ステータス表示はアイコン + テキスト併用

### エラーハンドリング

#### バリデーションエラー
```
入力欄の下に赤字でエラーメッセージ表示
例: 「利用日は今日以降の日付を指定してください」
```

#### APIエラー
```
画面上部にトースト通知（自動で消える）
または、モーダル内にエラーメッセージボックス

例:
┌────────────────────────────────────┐
│ ⚠️ 予約の作成に失敗しました       │
│ 指定の日時は既に予約済みです       │
│                          [閉じる]  │
└────────────────────────────────────┘
```

#### ネットワークエラー
```
画面中央にエラー表示 + リトライボタン
```

### ローディング状態

#### ボタンローディング
```
[処理中...]  ← スピナーアイコン + 無効化
```

#### ページローディング
```
画面中央にスピナー表示
またはスケルトンスクリーン（カード・テーブルの骨組み）
```

#### データローディング
```
データ取得中: スケルトンスクリーン
初回ロード: ローディングスピナー
```

## UIライブラリの選定

### 推奨ライブラリ

#### 1. **Chakra UI**（推奨）
- **メリット**:
  - カスタマイズ性が非常に高い（テーマ拡張が容易）
  - アクセシビリティ対応が標準
  - TypeScript対応
  - コンポーネントAPIがシンプル
- **デメリット**:
  - 学習コスト（独自のスタイルシステム）
  - バンドルサイズがやや大きい

#### 2. **Ant Design**
- **メリット**:
  - エンタープライズ向けの充実したコンポーネント
  - カレンダー・テーブルが強力
  - 日本語ドキュメント充実
- **デメリット**:
  - デザインの独自性が強い（カスタマイズが必要）
  - バンドルサイズが大きい

#### 3. **Material-UI (MUI)**
- **メリット**:
  - 成熟したライブラリ
  - コンポーネントが豊富
  - コミュニティが大きい
- **デメリット**:
  - Material Designの印象が強い
  - カスタマイズがやや複雑

### 最終推奨: **Chakra UI**

ブランドカラーを活かしたオリジナルデザインを実現しやすく、アクセシビリティ対応も標準装備のため、Chakra UIを推奨します。

## 実装ガイドライン

### ディレクトリ構造（フロントエンド）

```
frontend/
├── public/
│   └── favicon.ico
├── src/
│   ├── components/         # 共通コンポーネント
│   │   ├── common/         # ボタン、カード、モーダルなど
│   │   ├── calendar/       # カレンダー関連
│   │   └── forms/          # フォーム関連
│   ├── pages/              # 画面コンポーネント
│   │   ├── auth/           # 認証画面
│   │   ├── customer/       # 顧客向け画面
│   │   └── admin/          # 管理者向け画面
│   ├── hooks/              # カスタムフック
│   ├── services/           # API通信
│   ├── stores/             # 状態管理
│   ├── theme/              # テーマ設定
│   ├── types/              # TypeScript型定義
│   ├── utils/              # ユーティリティ関数
│   ├── App.tsx
│   └── main.tsx
├── package.json
└── vite.config.ts
```

### コンポーネント設計原則

1. **単一責任**: 1コンポーネント1責任
2. **再利用性**: 汎用的なコンポーネントは`components/common/`に配置
3. **Props型定義**: すべてのPropsにTypeScript型を定義
4. **アクセシビリティ**: ARIA属性を適切に設定
5. **エラーハンドリング**: ErrorBoundaryでラップ

### 状態管理方針

- **グローバル状態**: 認証情報、ユーザー情報、スタジオ情報
- **ローカル状態**: フォーム入力、UI状態（モーダルの開閉など）
- **サーバー状態**: TanStack Query（React Query）でキャッシュ管理
